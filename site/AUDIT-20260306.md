# Site Audit — getmarkup.dev
**Date:** 2026-03-06
**Version:** v0.9.1 (pre-launch)
**Scope:** index.html, docs.html, privacy.html, changelog.html
**Auditor:** Claude (Sonnet 4.6)

---

## TLDR

The site is visually strong and technically honest. The core copy is clear. The main risks before launch are:

1. **Persona mismatch**: The landing page reads like it's for developers (CSS selectors in hero copy, Dev Mode UI in hero mock). Your primary ICP can't read CSS.
2. **Two factual errors** that could damage credibility if noticed (title says "AI brief", storage called "encrypted").
3. **Docs are incomplete for contributors** — the Contributing section is thin, missing setup steps, PR workflow, and test status.
4. **Stats are unsourced** — 36% and 68% are credibility risks on a product with a genuine personal story.

Everything else is fixable detail. The origin story and the brief output demo are the strongest sections.

## ERRORS (objective, just wrong)

### E1 — Title tag says "AI fix brief" [index.html, line 6]
`<title>Markup — Annotate any page. Generate an AI fix brief.</title>`

The brief is explicitly template-based, not AI-generated. The extension UI, the "Brief output" section, and the footer copy all say "No AI · No API." The title contradicts this.
**Fix:** "Generate a fix brief." or "Generate a structured fix brief."

### E2 — chrome.storage.local called "encrypted" [docs.html, storage section]
`Stored in chrome.storage.local (secure, persistent, encrypted)`

`chrome.storage.local` is **not encrypted**. Data is stored in SQLite on disk and is accessible to anyone with access to the Chrome profile directory. Only `chrome.storage.managed` and keychain-backed credentials have encryption guarantees.
**Fix:** Remove "encrypted" — "secure, persistent, sandboxed to the extension" is accurate.

### E3 — "Install the Wispr Flow" [docs.html, voice notes section]
`Install the Wispr Flow Chrome extension from the Chrome Web Store.`

"the Wispr Flow" is grammatically wrong. Wispr Flow is a proper noun.
**Fix:** "Install Wispr Flow from the Chrome Web Store."

### E4 — Nav "Docs" links to anchor, not docs page [index.html, nav]
`<a href="#docs-section" class="nav-link">Docs</a>`

This scrolls to a docs preview section at the bottom of the homepage. Visitors expecting to read docs land in a teaser, not actual documentation. From all other pages, there's no Docs link in nav at all.
**Fix:** `href="/docs"` on index.html. Add Docs to nav on privacy.html.

### E5 — "Using with Claude" / "Using with Cursor" nav links have no content [docs.html]
The docs sidebar (both the homepage preview and docs.html) shows:
- "Using with Claude"
- "Using with Cursor"

Neither section exists anywhere in docs.html. Users who click these see nothing.
**Fix:** Either add content for both sections (they should exist — this is the core use case), or remove the nav items until content is written.

### E6 — Footer brand "N.Levin" on index.html and privacy.html
The footer brand reads "N.Levin" with the footer-brand style, then "Built by Noah Levin" below it. Two competing brand signals in one footer — and "N.Levin" is not the product brand.
**Fix:** Change footer brand to "Markup" on index.html and privacy.html to match docs.html and changelog.html.

### E7 — Chrome Web Store URL has no app ID [all pages]
`https://chrome.google.com/webstore/detail/markup`

This URL format requires an app ID: `https://chrome.google.com/webstore/detail/[name]/[id]`. Without the ID it will 404. The modern URL format is also `https://chromewebstore.google.com/detail/[id]`.
**Note:** Can't fix until the extension is submitted and gets an ID. But all "Add to Chrome" buttons should not be live-linked until then, or should link to a pre-launch waitlist/notify page instead of a broken Chrome Store URL.

## COPY ISSUES

### C1 — Stats are unsourced [index.html, problem section]
The 36% and 68% statistics have no source attribution. These appear to be estimates or invented numbers. On a product built on a genuine personal story, fabricated stats undermine credibility more than they help — especially with a developer audience that will notice.
**Options:**
- Remove the stats entirely and replace with a direct quote or personal observation
- Replace with qualitative framing: "Most vibe coders skip QA" is harder to challenge than "36% do"
- Add a footnote/asterisk with honest sourcing if real data exists

### C2 — "1 tab" stat is structurally different from the others [index.html, stats grid]
The stat grid has 36% / 68% / "1 tab" — the first two are large percentage numbers, the third is a small number with a word. In the animated counter format this looks broken. The third stat is actually a product claim, not a statistic.
**Fix:** Reframe third stat as a product differentiator or replace the grid with a different component entirely.

### C3 — "The exact CSS selector" in Step 01 is dev-speak [index.html, how it works]
`"The sidebar shows the exact CSS selector so you never have to guess what you're pointing at."`

Your primary ICP (non-developer vibe coder) doesn't know what a CSS selector is and doesn't care. This copy speaks to your secondary ICP.
**Fix for P1:** "The sidebar shows exactly which element you've highlighted — no guessing, no typing."
**Or:** Make this section show the Simple Mode experience for P1, and note Dev Mode features separately.

### C4 — "Build what I didn't" is slightly off [index.html, OSS section]
This implies the product is unfinished in a way that's slightly dissonant with the launch positioning.
**Fix:** "Build what comes next." or "The rest is yours."

### C5 — Hero mock shows Dev Mode as default [index.html, hero visual]
The sidebar mockup in the hero shows type picker (BUG/DESIGN/COPY), severity picker (Critical/High/Medium/Low), and selectors. This is Dev Mode — not the default mode. New installs start in Simple Mode.
This matters for P1 targeting: a non-developer vibe coder looking at the hero mock sees interface elements they won't have, and might think the tool is more complicated than it is.
**Consider:** Show Simple Mode in the hero (or a clean note-taking state without all the type/severity controls visible). Dev Mode features can be called out lower on the page.

### C6 — "The Builder" persona is labeled wrong for P1 [index.html, personas]
Per CLAUDE.md, P1 is "Non-developer vibe coder. Cannot read CSS/HTML." But the persona card shows "SOLO DEVELOPER / FOUNDER." A non-developer vibe coder is not a developer.
The quote is right ("capture every issue by voice and ship a brief to Claude") but the role label undermines the targeting.
**Fix:** Change role to "NON-TECHNICAL FOUNDER / VIBE CODER" or similar.

### C7 — "Aish" is opaque to most visitors [index.html, origin card]
"I build educational technology using AI at Aish" — most visitors don't know what Aish is. Without context it could read as anything.
**Fix:** Either add a one-word gloss ("at Aish, a global education organization") or just say "at a global education organization" to keep it clean without requiring visitors to look it up.

## DOCS — CONTRIBUTOR COMPLETENESS

The docs are well-structured for users. For open source contributors, the "Contributing" section is thin. Here's what's missing:

### D1 — No prerequisites listed
The install section jumps straight to `git clone`. It should state upfront:
- Node.js version requirement (add `.nvmrc` to repo first)
- npm version
- Chrome 114+ (for Side Panel API)

### D2 — No PR workflow documented
Nothing about how to submit contributions:
- Fork vs branch
- Where to open issues
- PR expectations (one feature per PR, etc.)
- Where CONTRIBUTING.md lives (doesn't exist yet per CLAUDE.md pre-launch checklist)

### D3 — Test suite not mentioned
There are currently no automated tests. The docs don't mention this. A contributor opening the repo and running `npm test` will get an error or nothing. This should be acknowledged: "There are currently no automated tests. QA is manual — load unpacked, test the core loop."

### D4 — "Adding Note Types" instructions don't match actual code
The docs say `Update noteTypes array with { id, label, color }`. In the actual sidebar.js, note type rendering, storage, and brief generation are handled through conditional strings, not a neat array. This will mislead contributors.
**Fix:** Either implement the array approach to match docs, or document what the code actually does.

### D5 — No mention of message protocol between content.js and sidebar.js
This is the key architectural pattern in the codebase — every feature crosses this boundary. A contributor who doesn't understand `chrome.runtime.sendMessage` / `onMessage` will be blocked immediately.

### D6 — "Using with Claude" and "Using with Cursor" sections are empty
These are the primary use cases for the brief output and should be the most detailed docs. At minimum they need:
- Copy the brief → paste into Claude/Cursor
- Suggested prompt framing
- Example exchange

### D7 — No mention of the `ignoreNextDeselect` flag or other gotchas
The CLAUDE.md has a whole "Key Architectural Decisions" section with non-obvious patterns (`deactivateReset` vs `resetForm`, `window.__markupActive`, the Chrome focus boundary workaround for floating input). None of this is in the docs. Contributors hitting these will spend hours debugging.

### D8 — Storage section calls local storage "encrypted"
See E2 above. Repeated here because it's in the docs, not just internally wrong — it's a claim facing contributors who will know it's false.

## PERSONA TARGETING SUMMARY

| ICP | Represented? | Gap |
|---|---|---|
| P1: Non-dev vibe coder | Partial | Hero mock shows Dev Mode. Step 01 copy uses "CSS selector." Persona card says "developer." |
| P2: Solo dev who vibe codes | Well served | Selector copy, brief format, export options all hit this. |
| P3: Freelancer | Present but weak | PDF export is the main value prop for them. Not highlighted prominently. |

**Biggest miss**: P1 is your primary ICP and the sharpest pain point. The landing page currently speaks past them. A non-developer vibe coder who arrives at the hero and sees type pickers, severity levels, and "CSS selectors" may think this tool is too technical for them. The entire Simple Mode feature exists for exactly this reason — it should lead.

**Recommendation**: Split the How It Works section or hero into two paths. "Just browsing and noting? Try Simple Mode." / "Annotating for a dev handoff? Try Dev Mode." The product already supports this. The page should show it.

The privacy page is the strongest of the four. Honest, clear, no legalese. A few small items:

- **Future-proofing**: v3.0.0 roadmap includes backend sync. Add one line: "This policy applies to Markup's current offline architecture. Any future cloud features will be opt-in with separate, updated terms."
- **Version number is hardcoded** — "Markup v0.9.1" will be stale at v1.0.0. Should be a note to update rather than hardcoded.

## WHAT'S WORKING WELL

- **Origin story**: The "built with one arm" section is the most differentiated content on the page. Specific, personal, true. Don't touch it.
- **The brief output demo**: The animated reveal of the formatted brief is genuinely compelling. Shows the product without explaining it.
- **Privacy page**: Best-in-class for a small extension. No weasel words.
- **"No AI · No API · Runs entirely in your browser"**: This framing is a real differentiator, especially as AI fatigue grows. Lean into it more.
- **Competitive table**: Four columns now covers the gap properly. The localhost column alone disqualifies every named competitor.
- **The personas**: The "before Markup" pain descriptions are sharp and credible.
