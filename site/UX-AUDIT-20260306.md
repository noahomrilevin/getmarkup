# UX/UI Audit — getmarkup.dev
**Date:** 2026-03-06
**Scope:** index.html, docs.html, privacy.html, changelog.html
**Method:** Persona-by-persona flow analysis, UI pattern review, missing content audit
**Auditor:** Claude (Sonnet 4.6)

## THE THREE PERSONAS

| Persona | Shorthand | Trigger | Goal on site |
|---|---|---|---|
| P1 | Non-dev vibe coder | Heard about it on Twitter / PH, curious if it's too technical | "Is this for me?" → Install |
| P2 | Solo dev who vibe codes | Actively looking for a QA/annotation tool, probably Googled | Evaluate features, verify localhost support → Install |
| P3 | Freelancer reviewing deliverables | Found via P2 recommendation or searching for "review annotate client work" | Understand PDF/brief export → Install or bookmark |

## P1 FLOW: NON-DEV VIBE CODER

**Entry point:** Homepage from Twitter link or Product Hunt
**Emotional state:** Cautiously optimistic, fast to bail if it looks like a dev tool

### What they see first (above the fold)

**Headline:** "Review your build. Brief your AI. Ship the fix."
→ ✅ **Clear.** "Brief your AI" resonates. No code terminology.

**Sub-headline:** "Annotate any page — localhost, staging, or live — leave notes with your voice, generate a structured brief, and paste it into Claude or Cursor."
→ ⚠️ **Mixed.** "localhost" and "Cursor" are technical. P1 may not know what localhost is — they probably just use Vercel preview URLs. "Cursor" is dev-tool specific. The word "structured" is slightly corporate.
**Recommend:** "Annotate any page — your Vercel preview, staging, or live site — leave notes by voice or text, generate a brief, and paste it into Claude to fix everything at once."

**Hero sidebar mock:**
→ ⚠️ **Dev Mode shown.** Type picker (BUG / DESIGN / COPY), severity labels, CSS selector displayed in notes. P1 sees complexity they won't have in Simple Mode. This is the sharpest contradiction in the hero: the headline says "for anyone who builds" but the interface shown says "for developers."
**Recommend:** Show Simple Mode in the hero mock. Dev Mode features visible further down the page after establishing Simple Mode as the default.

**CTAs:**
→ ✅ "Add to Chrome →" is clear. "View on GitHub ↗" is secondary, appropriate.

### Problem section ("THE VIBE CODING QA CRISIS")

**3 stat cards:** "No tool for this" / "Lost in tab switching" / "One paste"
→ ✅ Much better than the unsourced percentages they replaced. The qualitative framing is credible.

**Sub-copy under "Not because they don't care":**
> "...the tools just weren't built for this workflow."
→ ✅ Honest and relatable.

**However:** The section header "THE VIBE CODING QA CRISIS" will not mean anything to P1. "Vibe coding" as a term is developer slang. P1 might describe themselves as someone who "builds apps with AI" or "uses Claude to build my startup" — they would not self-identify as a "vibe coder."
**Recommend:** Change subheading to something like "BUILDING WITH AI, REVIEWING BY HAND" or drop the jargon entirely.

### How It Works section (scroll-synced walkthrough)

**Step 01:** "Click any element on any page."
→ ✅ Clear. The body copy now says "The sidebar shows exactly which element you've highlighted" (previously said "CSS selector") — this is the right call for P1.

**Step 02:** "Leave a note. Type or speak."
→ ✅ Perfect for P1. Voice-first resonates.

**Step 03:** "Generate the brief. Paste into Claude."
→ ✅ Clear. P1 knows Claude.

**Issue:** The sidebar mock in this section shows Dev Mode (type picker, severity, selectors). Same problem as the hero.
**Recommend:** Show Simple Mode for the 3-step walkthrough. Dev Mode features can appear in a "Power user features" or "Developer mode" subsection lower on the page.

**Scroll sync:** Now fixed (scroll-event based). But a subtler UX issue: P1 is unlikely to scroll through a 3-step walkthrough if it doesn't feel rewarding. The sticky sidebar mock updating is a good hook — just make sure it's actually showing the Simple Mode state (note added, then brief appears) so P1 sees the full loop.

### Personas section

**P1 card (Vibe Coder / Non-Technical Founder):**
→ ✅ Role label updated from "SOLO DEVELOPER" to "VIBE CODER / NON-TECHNICAL FOUNDER." This is the right fix.
→ ✅ Pain description and quote are sharp and credible.

**Issue:** The persona cards show 3 ICPs in a grid. P1 may see the P2 card ("SOLO DEVELOPER") and feel this product is more for developers than for them — even though their card is first.
**Minor recommend:** Add a small visual indicator or order emphasis that P1 is the primary target.

### Docs teaser section

**Issue:** Heading says "Everything you need to **extend** Markup." The word "extend" is contributor/developer language. P1 doesn't want to extend Markup — they want to use it.
**Recommend:** Split or reframe: "Everything you need to **use** Markup." or make the teaser explicitly about the user guide, not the dev docs. The CONTRIBUTING group shouldn't dominate a section aimed at users.

### Footer

**"Built by Noah Levin — CPO at Aish, a global Jewish education organization — educator"**
→ ✅ Personal, specific, credible.

## P2 FLOW: SOLO DEV WHO VIBE CODES

**Entry point:** Searching "annotation tool localhost" or "review localhost annotations Chrome extension"
**Emotional state:** Skeptical, efficient. They've tried Jam and BugHerd and know they don't work on localhost.

### Above the fold

**Headline:** "Review your build. Brief your AI. Ship the fix."
→ ✅ Resonates. Developer framing.

**Sub-headline mentioning "localhost":**
→ ✅ This is the hook. P2 will stop at "localhost" because every other tool fails here.

**Hero sidebar mock (Dev Mode):**
→ ✅ For P2, seeing the type picker and severity levels is *good*. Dev Mode is exactly what they want to see.
→ **Conflict with P1 recommendation:** Showing Dev Mode in hero is right for P2 but wrong for P1. The correct fix is a two-mode hero or hero that defaults to Simple but shows the full feature set.

### Competitive table (How it works section)

**Current table:** Works on localhost? / Element annotation? / Screenshot capture? / Fix brief output?
→ ✅ Strong. The localhost column alone disqualifies Jam and BugHerd. P2 will immediately understand the value prop here.

**Missing column:** "Works on file:// URLs?" — a minor power-user differentiator, but not critical.

### How It Works — Step 01

**"The sidebar shows exactly which element you've highlighted — no guessing, no typing."**
→ ⚠️ For P2, this is slightly dumbed down. P2 *wants* to see the CSS selector mentioned. The current copy hides a feature that P2 cares about (exact selector for AI handoff).
**Recommend:** Keep the P1-friendly copy for Step 01, but add a secondary detail for P2: "(Dev Mode shows the exact CSS selector for each note — pinpoint precision for AI handoff.)"

### OSS section

→ ✅ Well-positioned. P2 cares about MIT license, extensibility, open source.

### Docs teaser

**BRIEF FORMAT group:** P2 wants to know what the brief output looks like before installing. The sample brief in the teaser (the code block) does this well.
→ ✅ Good. The format reference, "Using with Claude," "Using with VS Code / Cursor" links are exactly right for P2.

## P3 FLOW: FREELANCER / CLIENT REVIEWER

**Entry point:** Colleague or client share, or Google search for "annotate web pages feedback client"
**Emotional state:** Practical. Primary question: "Can I send this to a client? Does it look professional?"

### Key features P3 needs to see

1. **PDF export** — professional deliverable for clients
2. **Screenshot capture** — visual proof alongside notes
3. **Works on any URL** — they review live client sites
4. **No client account required** — client shouldn't have to install anything

### What the homepage shows P3

**PDF export** is mentioned in the exports table in the docs but is *not prominently featured on the homepage*. P3 has to get deep into How It Works or find the Exporting section in docs to confirm PDF exists.
→ 🔴 **Gap.** PDF export for client-ready briefs is the core P3 value prop. It's not visible at first glance.
**Recommend:** Add one sentence to the hero sub-text or the OSS section: "Export as PDF for client handoff." or add it to the stat cards / feature callouts.

**Screenshot capture** is mentioned in Step 02 but the sidebar mock doesn't clearly show a screenshot attached to a note.
→ ⚠️ Minor gap. The screenshots feature is visible in the How It Works section.

**P3 persona card** exists but is positioned third and has the weakest pain description. The value for P3 is client professionalism, not just "review deliverables."

## UI PATTERN ISSUES

### Issue UI-1: Docs teaser section heading is contributor-facing
**File:** index.html
**Text:** "Everything you need to **extend** Markup."
**Problem:** P1 doesn't want to extend Markup. The word "extend" frames this as developer/contributor content. The actual teaser content includes user-facing docs (Getting Started, annotating, brief format) mixed with contributor content.
**Fix:** Change to "Everything about how Markup works." or show the User Guide tab content in the teaser and label it "User Guide."

### Issue UI-2: "DEVELOPER DOCS" eyebrow on homepage teaser
**File:** index.html
**Text:** `DEVELOPER DOCS` eyebrow text above the docs teaser section
**Problem:** If P1 sees "DEVELOPER DOCS," they skip this section. It's not developer docs — it's the product documentation.
**Fix:** Change eyebrow to `DOCUMENTATION` or `PRODUCT DOCS`.

### Issue UI-3: Docs tab switcher — default state
**File:** docs.html
**Current:** "User Guide" is default. ✅ Correct.
**Issue:** When a user visits `/docs` from the homepage "View full documentation →" link, they land in User Guide mode. ✅ Correct.
**But:** When a user clicks "View on GitHub ↗" from the Contributing tab area (contributor banner), they're sent to GitHub — not to the Contributing tab directly. This is fine.

### Issue UI-4: No active state in homepage nav
**File:** index.html
**Problem:** The homepage nav has no way to indicate "you are here." On other pages, the nav currently has no active link styling either.
**Fix:** Add a subtle `.active` style to nav links and mark the current page. Minor but adds polish.

### Issue UI-5: Mobile — Stats grid is 1-column
**File:** index.html
**Current CSS:** `@media (max-width:768px) { .stats-grid { grid-template-columns: 1fr; } }`
**Problem:** The stats section has 3 cards. On mobile they stack vertically with 3px gap, creating a very tall block of similar-looking content.
**Fix:** Consider 2 columns on mobile (with the third below), or reduce card padding on mobile so they feel lighter.

### Issue UI-6: Docs page — content area max-width too narrow
**File:** docs.html
**CSS:** `.content { max-width: 900px; }`
**Issue:** On very wide screens, the sidebar + content fills only ~1120px of a 2560px screen. The rest is blank. This is a minor issue — most users will be on laptops — but worth noting.

### Issue UI-7: Brief typewriter animation timing
**File:** index.html
**Current:** Brief lines appear sequentially with CSS class transitions triggered by IntersectionObserver.
**Issue:** If a user scrolls past the brief section quickly, the animation may not complete. Also, the "COPY TO CLIPBOARD" button is live — copying the sample brief is a nice interactive touch, but there's no confirmation feedback beyond the button text changing. ✅ Actually works.

### Issue UI-8: Persona card "before Markup" pain descriptions are sharp but the "after" is implied, not shown
**File:** index.html
**Issue:** P1 card says "I can't tell where my feedback actually lands when I give it to Claude" — excellent. But there's no explicit "after" moment on the page that shows: "with Markup, Claude fixes exactly the right element."
**Fix:** The brief output demo is the "after" — but there's no visual connection between the persona pain and the brief output. A short quote or outcome statement below each persona card would close the loop.

## WHAT'S WORKING WELL

- **Brief output demo** is the strongest section on the page. The sequential line reveal, the copy button, the format reference — all of it communicates the product faster than any copy.
- **Competitive table** is clean and decisive. The localhost column alone is the entire product positioning.
- **Privacy page** is genuinely excellent. Short, honest, no weasel words. Rare.
- **Origin story** ("built with one arm") is the most human content on the site. Don't dilute it.
- **Changelog** has real dates and sprint-by-sprint progress — good for the audience who cares (developers, PH voters).
- **Docs tab switcher** (just implemented): clean UX that doesn't penalize users who only want the user guide.
- **"No AI · No API · Runs entirely in your browser"** is a real differentiator. It appears in the brief teaser but should be more prominent — possibly in the hero or as a second line in the nav CTA.

## SUMMARY JUDGMENT

The site is close to launch-ready. The visual design is strong, the content is honest, and the core value prop (localhost annotation → AI-ready brief) is communicated clearly. The two structural risks before launch:

1. **P1 sees a developer interface in the hero and leaves.** Showing Dev Mode in the hero is the single biggest persona mismatch. The Simple Mode feature was built specifically for P1 — it should lead.

2. **PDF export is invisible to P3.** Adding one mention ("Export as PDF for client handoff") in the hero sub-text or feature section closes this gap completely.

Everything else is incremental. The origin story, the brief output demo, and the competitive table are already excellent. Ship those; fix the hero mock and PDF mention; address the Chrome Store ID when the extension is submitted.
